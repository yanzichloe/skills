## Overview

qieman h5 design 负责营销 H5 的页面逻辑、内容框架、视觉氛围、转化路径与营销 Pattern。生成营销 H5 时，应先遵守 **L0 `qieman-design-ui`**，再叠加本 **L2** 规范（分层说明见 `../qieman-design-ui/SKILL.md`）。

---

## 1. Skill Overview

### 1.1 适用范围

qieman h5 design 适用于：

- 且慢 App 内营销 H5
- 策略介绍页
- 活动营销页
- 优惠券 / 福利页
- 节假日活动页
- 定投教育 / 定投活动页
- 账户诊断 / 持仓分析页
- 权益 / 会员页
- 新手指南 / App 使用指南
- AI / 产品功能上线页
- 市场解读 / 投教专题页
- 年度账单 / 半年度报告页
- 基金品类投资指南页
- 服务信任 / 品牌陪伴页

### 1.2 不适用范围

不适用于：

- PC 官网整站
- 后台管理系统
- 纯交易表单页
- 复杂数据分析看板
- PPT / Keynote 报告页
- 非且慢品牌视觉页面

### 1.3 与规范分层的关系

本 skill 属于 **L2 独立场景**（见 `../qieman-design-ui/SKILL.md` → Skill Layering）：

- **L0 `qieman-design-ui`**：品牌色、字体、按钮、卡片、App 容器、常规交互。
- **L2 `qieman-design-h5`（本规范）**：营销 H5 页面逻辑、内容框架、视觉氛围、转化路径与 Pattern。
- **L1 `qieman-design-sell-popup`**：若 H5 内嵌卖出挽留弹窗，另读 [`../qieman-design-ui/references/qieman-design-sell-popup.md`](../qieman-design-ui/references/qieman-design-sell-popup.md)。

生成营销 H5 时：**先 L0，再本 L2**；含挽留弹窗时 **再叠 L1**。

## 2. Core Brand Principles

### 2.1 金融可信

营销页必须建立信任感，而不是单纯制造刺激感。

要求：

- 收益、回撤、胜率等数据必须有来源和口径。
- 不得承诺收益、保本、稳赚。
- 风险提示必须保留且可读。
- 图表和数据必须克制、清楚、移动端可读。

### 2.2 克制轻盈

且慢营销页可以有活动感，但不能电商化。

避免：

- 大红大金
- 红包雨
- 夸张金币堆叠
- 强刺激促销语
- “错过血亏”“稳赚不赔”等焦虑表达

### 2.3 用户陪伴

页面语气应像专业顾问陪伴用户理解问题。

推荐：

- 安抚而不煽动
- 专业但不生硬
- 有解释感
- 有长期主义
- 有行动引导

### 2.4 数据清晰

所有图表、表格和数据卡必须服务一个明确结论。

要求：

- 每张图只表达一个核心观点。
- 数据卡不堆满指标。
- 图表下方必须有数据来源、统计时间或说明。
- 复杂内容应被拆成卡片、步骤或图示。

### 2.5 移动优先

所有页面默认在且慢 App WebView 中阅读。

- 设计基准宽度：375px
- 高度不限
- 单列纵向阅读
- 禁止横向滚动
- 内容卡默认宽度：343px
- 左右安全边距：16px

## 3. App H5 Canvas Rules

### 3.1 页面容器

```css
html,
body {
  margin: 0;
  width: 100%;
  min-height: 100%;
  overflow-x: hidden;
  font-family: "PingFang SC", "MiSans", system-ui, -apple-system, sans-serif;
  color: #333333;
}

body {
  display: flex;
  justify-content: center;
  background: #E8F1FE;
}

.phone,
.page {
  width: 100%;
  max-width: 375px;
  min-height: 100vh;
  margin: 0 auto;
  overflow: hidden;
  background: var(--surface-page, #ECF4FF);
  position: relative;
}
```

### 3.2 App Shell 强约束

- 顶部导航、内容区、底部 CTA 必须属于同一个 375px 容器。
- 不允许导航栏全浏览器宽度、内容区 375px 的割裂结构。
- 不允许 PC 布局混入 App H5。
- 页面可以在浏览器中居中显示，但 App 页面主体必须完整。

推荐结构：

```html
<body>
  <main class="phone">
    <header class="statusbar"></header>
    <nav class="topnav"></nav>
    <section class="hero"></section>
    <div class="content"></div>
    <section class="risk"></section>
    <footer class="fixed-cta"></footer>
  </main>
</body>
```

### 3.3 内容宽度

- 页面容器：375px
- 内容卡片：343px
- 页面左右边距：16px
- 内容卡高度：不限制，按内容自适应
- 风险提示可使用 343px 内容宽度，也可作为正文底部弱化文本区

## 3.8 Output Format Rules

默认输出格式必须为 **HTML 单文件**。

除非用户明确要求其他格式，否则所有营销 H5、策略介绍页、活动页、功能页、报告页、App WebView 页面设计，都应输出为可直接打开和验证的 `.html` 文件。

### 1. 默认交付物

默认交付物：

```txt
.html
```

要求：

- 输出完整 HTML 文档。
- 必须包含 `<!DOCTYPE html>`、`<html>`、`<head>`、`<body>`。
- CSS 写在 `<style>` 内。
- JS 如有需要，写在 `<script>` 内。
- 页面应可直接在浏览器、本地文件、Cursor Preview、Figma html.to.design 等场景打开。
- 不默认输出 React、Vue、Svelte、小程序代码或 Markdown 方案稿。
- 不只输出局部代码片段。

### 2. 单文件原则

生成的 HTML 应尽量为单文件：

- HTML + CSS + 必要 JS 合并在一个文件中。
- 图标优先使用 inline SVG 或 Remix Icon 类名 + 降级方案。
- 关键插图、图表、装饰元素优先使用 inline SVG / CSS 绘制。
- 不依赖外部图片作为唯一视觉来源。
- 不依赖外部 CDN 作为关键内容展示的唯一方案。

允许：

- 使用 CSS 渐变、SVG、HTML 卡片、伪元素绘制氛围图。
- 使用少量 JS 实现 tab、展开收起、锚点滚动等交互。
- 如果必须使用外部资源，需要提供本地可展示的替代方案。

禁止：

- 只输出设计说明不输出 HTML。
- 只输出 CSS 不输出页面结构。
- 只输出伪代码。
- 输出需要复杂构建流程才能运行的项目。
- 依赖 npm install、webpack、vite 等构建工具作为唯一运行方式。

### 3. 文件命名规则

当生成文件时，推荐使用清晰的英文或拼音命名：

```txt
qieman_strategy_intro.html
qieman_campaign_h5.html
qieman_account_diagnosis.html
shouzhengchuqi_strategy_h5.html
```

命名要求：

- 小写字母
- 使用 `-` 或 `_`
- 包含页面主题
- 文件后缀必须为 `.html`

### 4. 页面尺寸规则

HTML 页面默认按 App WebView 设计：

```css
.page {
  width: 375px;
  min-height: 100vh;
  margin: 0 auto;
}
```

要求：

- 默认设计宽度为 375px。
- 页面内容区按移动端单列布局。
- 主卡片宽度通常为 343px。
- 高度不限，根据内容自然延展。
- 页面可在桌面浏览器中居中预览。
- 不做 PC 宽屏布局，除非用户明确要求。

### 5. HTML 结构建议

推荐基础结构：

```html
<!DOCTYPE html>
<html lang="zh-CN">
<head>
  <meta charset="UTF-8" />
  <meta name="viewport" content="width=device-width, initial-scale=1.0, maximum-scale=1.0" />
  <title>且慢营销 H5</title>
  <style>
    :root {
      --brand-blue: #1B88EE;
      --surface-page: #F0F6FF;
      --text-main: #333333;
      --text-sub: #606060;
      --text-light: #999999;
      --card-white: #FFFFFF;
    }

    * {
      box-sizing: border-box;
    }

    body {
      margin: 0;
      font-family: -apple-system, BlinkMacSystemFont, "PingFang SC", "Microsoft YaHei", sans-serif;
      background: #EEF4FF;
      color: var(--text-main);
    }

    .page {
      width: 375px;
      min-height: 100vh;
      margin: 0 auto;
      background: var(--surface-page);
      overflow: hidden;
    }
  </style>
</head>
<body>
  <main class="page">
    <!-- Hero / Content / Risk / CTA -->
  </main>
</body>
</html>
```

### 6. 可交互要求

如果页面包含交互，应使用原生 HTML/CSS/JS 实现：

- tab 切换
- 展开收起
- FAQ
- 锚点跳转
- 按钮状态
- 规则弹层
- 简单滑动卡片

要求：

- 交互代码简洁。
- 不依赖框架。
- 不影响静态展示。
- 交互失败时页面主体仍可阅读。

### 7. 视觉资源实现要求

由于默认输出为 HTML，视觉资源应优先采用可代码化方式实现：

- Hero 氛围图：CSS 渐变 + inline SVG + 伪元素
- 图标：Remix Icon / inline SVG
- 图表：inline SVG / HTML + CSS
- 卡片：HTML + CSS
- 装饰光影：CSS radial-gradient / linear-gradient
- 背景纹理：CSS / SVG

禁止：

- 用无法访问的远程图片链接作为主视觉。
- 使用空白图片占位。
- 用 `img src="xxx.png"` 指向不存在的本地文件。
- 页面视觉依赖用户手动补图才能成立。

### 8. 输出验收清单

交付 HTML 前必须检查：

- 是否是完整 `.html` 单文件？
- 是否能直接打开？
- 是否包含完整 head / style / body？
- 是否输出完整 HTML 单文件，而不是局部代码或说明稿？
- 是否为 375px 移动端页面？
- 是否包含 Hero、内容卡、风险提示、CTA 等必要结构？
- 关键图标 / 图表 / 插图是否不会因外部资源失败而空白？
- 是否没有构建依赖？
- 是否符合且慢品牌视觉？
- 是否适合导入 Figma 后继续编辑？

## 4. Marketing Page Design Framework

### 4.1 Page Type First

生成营销 H5 前，必须先识别页面类型，并选择对应 Pattern。  
不得在未判断页面类型的情况下直接生成通用营销页。

必须判断：

- 页面是策略介绍、活动页、功能上线、定投教育、诊断转化、权益说明、报告复盘，还是投教专题？
- 页面唯一主目标是什么？
- 用户当前处于什么情境？
- 页面最终希望用户做什么？

### 4.2 One Page, One Main Goal

每个页面只能有一个主目标。

示例：

- 优惠券页：领取 / 使用优惠券
- 诊断页：添加顾问开启分析
- 定投页：开启定投
- 功能上线页：体验新功能
- 报告页：建立信任并继续持有
- 权益页：去达标 / 去领取权益

### 4.3 Scene → Value → Proof → Action

营销页通用递进链路：

```txt
场景切入
→ 用户问题 / 机会
→ 解决方案
→ 价值证明
→ 操作路径
→ 行动按钮
→ 风险提示
```

不同 Pattern 可以变化顺序，但不得缺少理解路径。

### 4.4 One Screen, One Key Message

移动端每一屏只表达一个核心观点。

要求：

- 一张卡片只表达一个核心信息。
- 长段落必须拆成短段、列表、图表或步骤。
- 不要在一个模块内同时讲背景、规则、收益、操作、风险。

### 4.5 User Journey First

先设计用户理解路径，不直接堆内容。

必须回答：

1. 用户为什么要看？
2. 用户当前遇到什么问题？
3. 且慢提供什么解决方式？
4. 为什么这个方式可信？
5. 用户下一步该做什么？
6. 风险与规则是什么？

### 4.6 Visual Trust over Promotion

且慢营销页优先建立信任感，不做强刺激促销。

可以有：

- 轻活动感
- 情绪承接
- 福利点
- 节日氛围
- 成就感
- 陪伴感

但不能做成：

- 电商大促页
- 红包抽奖页
- 暴富暗示页
- 夸张收益承诺页

### 4.7 Data Must Have Context

所有收益、回撤、胜率、用户占比、历史表现必须有来源和口径。

要求：

- 数据来源
- 统计时间
- 历史测算 / 实盘说明
- 不代表未来表现说明
- 必要时补充适当性说明

### 4.8 CTA Must Match Page Goal

CTA 必须与页面主目标一致。

推荐：

- 去领取
- 去充值
- 去定投
- 去看看
- 开启信号定投
- 添加顾问开启分析
- 体验智能转换
- 开启转托管

避免：

- 泛泛写“立即购买”
- 一个页面多个同权重 CTA
- CTA 与页面内容无关

### 4.9 Risk Disclosure Always Visible

风险提示必须保留且可读，不得隐藏到不可见位置。

## 4.10 Marketing H5 Structural Skeleton

且慢营销 H5 默认采用：

```txt
Hero 主题氛围区
↓
Content 主内容卡片区
↓
Risk Disclosure 风险提示区
↓
Bottom CTA 底部行动区
```

### 4.10.1 Hero 主题氛围区

Hero 用于承接主题、建立情绪、突出主利益点。

默认规则：

- 宽度：375px
- 推荐高度：320px–420px，常规约 400px
- 包含页面标题、副标题、主视觉、核心标签或关键权益
- Hero 不能过高，避免挤压首屏关键信息
- Hero 与下方内容区必须自然过渡，不能硬切

### 4.10.2 Content 主内容区

主内容区默认采用居中的卡片承载信息。

规则：

- 内容卡宽度：343px
- 左右边距：16px
- 卡片高度不限
- 卡片之间需保持稳定垂直间距
- 主内容区采用单列纵向阅读流

常见卡片：

- 数据卡
- 说明卡
- 图表卡
- 步骤卡
- 权益卡
- 策略卡
- 顾问服务卡
- 活动规则卡

### 4.10.3 Content Stacking Rules

- 每个模块独立成卡。
- 一张卡只表达一个核心信息。
- 内容按“背景 / 价值 / 证明 / 操作 / 承接”递进。
- 长文必须拆卡，不直接堆成长段落。

### 4.10.4 Risk Disclosure Placement

风险提示应位于主要内容之后、底部 CTA 之前或页面底部区域中。

要求：

- 与主内容视觉层级拉开。
- 可采用纯文本区，不强制白卡。
- 字号应保证可读。
- 不得折叠、隐藏或弱化到不可阅读。

### 4.10.5 Bottom CTA

页面底部应设置清晰主 CTA。

要求：

- 按钮居中。
- 使用且慢品牌蓝。
- 文案表达明确动作。
- CTA 与页面唯一主目标一致。
- 次级 CTA 不应抢主 CTA 权重。

### 4.10.6 Layout Type A / Type B

#### A. 首屏氛围图 + 下方卡片流

适合：

- 策略介绍页
- 活动营销页
- 优惠券页
- 假期收益页
- 功能上线页

结构：

```txt
Hero 氛围区（约 375×400）
↓
343px 卡片流
↓
风险提示
↓
CTA
```

#### B. 长内容专题 / 长文解释页

适合：

- 服务信任页
- 市场解读页
- 品类指南页
- 年度报告页
- 复盘页

结构：

```txt
简化 Hero / 标题区
↓
长内容卡片流
↓
图表 / 案例 / 说明块
↓
风险提示 / 数据口径
↓
CTA（可弱化）
```

### 4.10.7 Structural Priority Rule

页面结构优先级高于装饰表现。

先确定：

- Hero 区
- 内容卡片区
- 风险提示区
- CTA 区

再决定插画、颜色、图标和图表。  
不得只追求局部视觉效果而破坏整体阅读结构。

## 5. Base Information Architecture

### 5.1 Hero 首屏

Hero 应快速回答：

- 这是什么页面？
- 和我有什么关系？
- 我为什么要继续看？

Hero 常见内容：

- 主题 / 场景 / 主利益点
- 一句话价值主张
- 主视觉
- 首屏关键数据或关键权益

### 5.2 Context 背景说明

用于解释用户为什么需要关注。

可包含：

- 市场背景
- 活动背景
- 用户痛点
- 机会窗口
- 政策 / 节假日 / 额度变化

### 5.3 Core Value 核心价值

根据页面类型表达：

- 策略价值
- 产品价值
- 权益价值
- 功能价值
- 服务价值
- 配置价值

### 5.4 Evidence 证明模块

可使用：

- 数据
- 案例
- 图表
- 对比
- 历史验证
- 用户行为
- 主理人动作
- 服务记录

### 5.5 How-to 操作路径

用于降低行动门槛。

可包含：

- 领取方式
- 使用步骤
- 开通流程
- 定投方式
- 联系顾问
- 操作截图
- 时间轴 / 步骤流

### 5.6 Product / Service Carrying 承接模块

可承接到：

- 策略卡
- 产品卡
- 顾问服务
- 活动入口
- AI / 功能入口
- 配置专区
- 会员权益

### 5.7 CTA 行动转化

CTA 类型：

- 主按钮
- 次按钮
- 固定底部按钮
- 模块内按钮

CTA 必须与页面主目标一致。

### 5.8 Risk / Rules 风险与规则

必须根据页面类型补充：

- 投资风险
- 活动规则
- 数据口径
- 适当性说明
- 功能规则
- 优惠券限制
- 统计方法

## 6.1 Color System for Marketing H5

也就是：

```txt
+ 蓝色行动按钮
```

### 6.1.1 Primary Color Rules

蓝色是品牌可信主色。

推荐用于：

- 主 CTA
- 关键操作按钮
- 功能入口
- 重要标签
- 图表主线
- 金融科技感主视觉


### 6.1.2 Background Color Rules

页面背景不建议通屏纯白，应使用低饱和浅色氛围底。

按场景：

- 策略 / 功能 / 教育类：浅蓝、浅紫蓝
- 活动 / 权益 / 节日类：浅暖色、浅粉、浅橙
- 报告 / 复盘类：浅蓝灰、浅紫蓝
- 诊断 / 工具类：浅蓝灰、科技浅蓝

推荐：

避免：

- 大面积纯黑
- 高饱和大红
- 高饱和金色
- 大面积深蓝压迫感
- 电商促销撞色

### 6.1.3 Card Color Rules

正文内容必须优先使用白色或近白色卡片承载。

推荐：

```css
--card-main: #FFFFFF;
--card-soft: #F8FBFF;
--card-warm: #FFF8F3;
--card-blue: #F0F7FF;
--card-purple: #F4F3FF;
```

要求：

- 卡片与页面背景有轻微层级差。
- 不使用重描边。
- 不使用高对比强阴影。
- 保证文字可读。

### 6.1.4 Warm Accent Rules

- 优惠券金额

- 截止提醒
- 关键高亮词

```css
--benefit-orange: #FF7A45;
--soft-red: #FA6A55;
--risk-red: #FA440C;
--warning: #EA9500;
```

避免：

- 大面积红色背景
- 红金组合
- 红包雨
- 过度刺激的抢购感

### 6.1.5 Scenario-based Color Palettes

#### 策略介绍页

```txt
浅蓝 / 浅紫蓝 + 白卡 + 品牌蓝 CTA + 少量橙红数据
```

气质：专业、克制、长期、可信。

#### 活动福利页

```txt
浅蓝 / 浅暖色渐变 + 白卡 + 暖色利益点 + 蓝色 CTA
```

气质：有活动感，但不电商化。

#### 节日活动页

```txt
浅橙 / 浅粉 / 米白 + 少量节日暖色 + 蓝色 CTA
```

气质：温暖、有节日感，但不俗气。

#### 定投教育页

```txt
浅蓝绿 / 浅蓝 + 自然色点缀 + 白卡 + 蓝色 CTA
```

气质：长期、生长、陪伴、规律。

#### 报告 / 年度账单页

```txt
浅蓝紫 + 白卡 + 少量金色 / 橙色成就点缀
```

气质：仪式感、可信、复盘感。

#### 功能上线页

```txt
科技浅蓝 + 白卡 + 蓝色按钮 + 少量橙色高亮操作点
```

气质：产品感、效率感、智能感。

#### 权益 / 会员页

```txt
浅蓝 + 浅紫 + 少量金色 / 珠光色
```

气质：专属感、礼遇感，但不过度奢华。

#### 诊断 / 工具页

```txt
浅蓝灰 + 白卡 + 数据图表蓝 + 少量风险橙红
```

气质：专业、分析感、可信。

### 6.1.6 CTA Color Rule

无论页面主题色如何变化，主 CTA 默认使用且慢品牌蓝。

原因：

- 保持 App 内操作一致性
- 降低金融交易动作的不确定感
- 避免活动页过度促销化

除非明确要求，不使用红色、橙色、金色作为主 CTA。

## 6.4 Hero Image Rules

### 6.4.1 完整氛围图

策略页、活动页、专题页的首屏氛围图必须完整，不允许像 CSS 几何拼图一样破碎。

要求：

- 统一天空 / 背景 / 主视觉关系
- 不出现断层、错位、重复、残缺
- 图像底部与页面背景自然衔接

### 6.4.2 Hero Transition Rules

头图底部必须与页面背景或首张卡片自然过渡。

推荐方式：

- 渐隐过渡
- 雾化过渡
- 首卡上浮
- 背景延续
- 地形 / 水面延展

推荐 CSS：

```css
.hero::after {
  content: "";
  position: absolute;
  left: 0;
  right: 0;
  bottom: 0;
  height: 72px;
  background: linear-gradient(
    180deg,
    rgba(236,244,255,0) 0%,
    rgba(236,244,255,.76) 62%,
    var(--surface-page) 100%
  );
  pointer-events: none;
}
```

### 6.4.3 首卡上浮

- 首张数据卡 / 权益卡可上浮覆盖头图底部。
- 上浮高度建议：16px–32px。
- 首卡圆角建议：16px–20px。
- 首卡与头图之间应是“承接”，不是“断开”。

### 6.4.4 Hero Layered Gradient Rules

为增强首屏视觉层次感，头部氛围图不得只使用单一浅色填充。Hero 需要通过深浅渐变、前中景层次和底部过渡建立空间感。

#### 渐变层次

推荐使用纵向渐变：

- 顶部：较深品牌蓝或主题蓝，例如 `#2F7BEC` / `#2679EE`
- 中段：柔和浅蓝，例如 `#BFDFFF`
- 底部：接近页面背景色，例如 `#ECF4FF` / `#F0F6FF`

要求：

- 顶部与底部必须有明确明度差。
- 渐变应柔和，不使用强烈断层。
- 不得整块使用过浅的单色背景，导致首屏缺乏层次。
- 活动类页面可使用浅暖色渐变，但仍需有前后层次和光影变化。

#### 前中景元素

Hero 可增加前景 / 中景 / 远景层次：

- 山、水、道路、地平线、日出、云雾
- 数据面板、产品卡片、金融装置
- 人物、顾问、AI 助手
- 光线、投影、漂浮小元素

要求：

- 前景元素饱和度可比背景高 5%–15%。
- 主视觉应与页面主题相关。
- 不得只放一个过小、过淡、无主题关联的装饰元素。
- 前中景元素必须服务标题和页面气质，不能抢正文。

#### 底部过渡

Hero 底部必须与首张卡片或页面背景自然衔接。

推荐：

- 雾化过渡
- 渐隐遮罩
- 光影延展
- 首卡上浮
- 地形 / 水面 / 色块延展

禁止：

- 硬切断层
- 直接拼贴
- 背景图底部突然结束
- 首卡与头图之间没有视觉承接

---

### 6.4.5 Hero Visual Type Judgment

生成首屏主视觉前，必须先判断主视觉类型，再决定表现方式。

#### 图标型主视觉

适用：

- 单个产品图标
- 功能图标
- 小型金融组件
- 轻 3D 装置
- 数据面板 / 卡片 / 小型图表装置

规则：

- 图标造型必须完整、清晰、精细。
- 应具备一定 2.5D / 轻 3D 质感。
- 结构简洁，不堆叠过多元素。
- 与页面主题强相关，例如收益、资产配置、账户、图表、权益、功能。
- 与参考图构图尽量接近。
- 图标与背景之间要有呼吸感，不要过小、过淡或漂浮失衡。
- 右侧小图标必须像“页面主题符号”，不能像临时占位图。

推荐方向：

- 圆角卡片 / 面板 / 数据小组件
- 轻拟物金融图表
- 精细化功能 icon
- 少量高光、阴影、玻璃感、描边增强精致度

避免：

- 粗糙占位图标
- 简笔画式低完成度图标
- 元素比例失真
- 图标主题与页面文案不一致
- 图标细节过少，看起来像临时拼出来

#### 风景 / 氛围型主视觉

适用：

- 山、水、道路、日出、海面、天空
- 策略意境图
- 品牌情绪化氛围图
- 长期、稳健、陪伴、安心类策略页

规则：

- 必须尽量贴近参考图的构图关系和情绪表达。
- 优先复现参考图的主体关系：山形、水面、道路、日出、光线、远山层次。
- 画面应简洁、完整、低饱和。
- 风景层次要清楚：前景 / 中景 / 远景。
- 视觉重点应服务标题气质，例如稳、安、长期、陪伴。
- 不要把参考中的大场景缩成角落里的小装饰。
- 不要随意替换成无关产品图标。

避免：

- 只保留一个模糊色块
- 只画一个孤立山包
- 用抽象几何替代应有的风景关系
- 场景比例与参考差异过大
- 风景型页面误生成成产品小图标

#### 类型不可混淆

强约束：

- 图标型页面不要误生成复杂风景主体。
- 风景型页面不要误生成过于产品化的小图标。
- 当用户提供参考图时，必须优先判断参考图是图标型还是风景型。

---

### 6.4.6 Reference-based Hero Fidelity Rules

当存在明确参考图、历史样例或用户指定方向时，首屏主视觉必须优先贴近参考图，而不是自由发挥。

必须优先还原：

1. 主体类型
2. 构图位置
3. 元素比例
4. 风格气质
5. 精细程度
6. 明暗层次
7. 与标题和卡片的空间关系

允许变化：

- 细节优化
- 配色微调
- 质感提升
- 清晰度提升
- 轻微品牌化处理

不允许变化过大：

- 主视觉类型被替换
- 构图关系明显跑偏
- 从图标页变成风景页
- 从风景页变成产品图标页
- 参考中的核心元素被弱化为无关装饰
- 右侧视觉锚点变成粗糙占位图

---

### 6.4.7 Small Hero Anchor Visual Rules

当首屏采用“左侧标题 + 右侧小主视觉”的布局时，右侧主视觉必须具备足够精细度和识别性。

规则：

- 位置一般位于右上区域，与左侧标题形成视觉平衡。
- 尺寸不宜过小，应能明确识别主题。
- 视觉应像“页面主题图标”或“页面主题符号”，不是随意装饰。
- 图标需要有精细结构、轻阴影、体积感或玻璃质感。
- 不得使用过淡、过小、过空的图形导致页面右侧失衡。
- 如果参考图为小型 3D 图标，应尽量还原图标的透视、圆角、光影和主题元素。

推荐：

- 精细化小图标
- 小型 3D 装置
- 小型数据面板
- 小型场景化金融组件
- 产品功能符号

---

### 6.4.8 Hero Visual Acceptance Checklist

生成或验收 Hero 主视觉时，必须检查：

- 是否先判断了主视觉类型？
- 图标型主视觉是否足够精细？
- 风景型主视觉是否贴近参考图构图？
- 右侧小主视觉是否与标题形成平衡？
- 主视觉是否与页面主题强相关？
- 头部是否有足够明暗层次？
- Hero 底部是否与首卡自然过渡？
- 是否存在明显占位感、拼贴感、粗糙感？

### 6.4.9 Strategy Hero Atmosphere Rules

策略介绍页的 Hero 不应只是“浅蓝渐变 + 抽象波浪 + 孤立 icon”的拼接，而应构建一张完整的营销氛围图。

#### 1. Hero 必须具备完整场景结构

策略页 Hero 至少应包含三层空间关系：

1. 远景层  
   - 天空
   - 远山
   - 云雾
   - 日出 / 柔和光源
   - 渐变背景

2. 中景层  
   - 水面
   - 蜿蜒路径
   - 山脊线
   - 地平线
   - 柔和曲线

3. 前景层  
   - 策略 icon
   - 信号面板
   - 资产配置卡
   - 数据小组件
   - 投顾 / 顾问人物

要求：

- Hero 必须看起来像一张完整的营销氛围图，而不是 UI 背景色块拼接。
- 场景元素之间需要有空间关系，不能只是装饰元素堆叠。
- 标题、主视觉、首张卡片之间要形成完整视觉路径。

#### 2. 策略页 Hero 应有明确光源

Hero 背景必须有柔和光源，增强空间纵深。

推荐：

- 光源位于右上、左上或远山后方。
- 使用浅蓝白光或淡暖光。
- 形成从亮部到暗部的自然层次。

推荐色彩：

```css
--hero-top: #2F7BEC;
--hero-mid: #86BDF8;
--hero-bottom: #EAF4FF;
--hero-light: rgba(255,255,255,.55);
--hero-warm-light: rgba(255,232,190,.25);
```

禁止：

- 整块单一浅蓝背景。
- 没有光源变化。
- 背景过浅导致层次不足。
- 背景过深影响标题可读性。

#### 3. 抽象波浪必须转译为场景元素

策略页 Hero 中的抽象波浪，不应只是装饰色块，应尽量转译为具备场景语义的元素。

推荐转译为：

- 远山轮廓
- 水面层次
- 蜿蜒道路
- 云雾层
- 山谷地形
- 地平线

禁止：

- 无意义的抽象弧线堆叠。
- 生硬波浪色块。
- 与主题无关的背景装饰。
- 只有色块，没有场景语义。

#### 4. 右侧 icon 必须成为前景锚点

当 Hero 使用“左侧标题 + 右侧小主视觉”布局时，右侧 icon 不应像孤立贴图。

要求：

- 右侧 icon 应作为前景锚点。
- 与背景有轻微投影、光晕或落点关系。
- 体积建议占 Hero 宽度的 22%–30%。
- 不要超过标题视觉权重。
- 必须与策略主题强相关。

适合守正出奇这类策略页的 icon：

- 策略信号面板
- 上升曲线 + 节奏点
- 资产配置小卡
- 罗盘 / 路径 / 指针
- 全球配置小球 + 数据面板

禁止：

- 粗糙占位 icon。
- 与页面主题无关的图标。
- 过小、过淡、漂浮失衡。
- 看起来像临时贴上去的装饰图。

#### 5. Hero 与首张卡片必须自然衔接

Hero 底部必须和首张内容卡片形成视觉承接。

推荐方式：

- Hero 底部渐隐到页面背景。
- 首张白卡上浮 16px–32px。
- 首卡带轻微阴影。
- 水面 / 路径 / 云雾延展到首卡背后。
- Hero 底部使用柔和雾化遮罩。

推荐 CSS：

```css
.hero::after {
  content: "";
  position: absolute;
  left: 0;
  right: 0;
  bottom: 0;
  height: 88px;
  background: linear-gradient(
    180deg,
    rgba(234, 244, 255, 0) 0%,
    rgba(234, 244, 255, .72) 58%,
    var(--surface-page, #ECF4FF) 100%
  );
  pointer-events: none;
}

.first-card {
  position: relative;
  margin-top: -24px;
  z-index: 2;
}
```

禁止：

- Hero 底部硬切。
- 首卡和背景完全断开。
- 背景图突然结束。
- 首卡没有上浮或承接关系。

#### 6. 不同策略气质对应不同 Hero 场景

##### 稳健 / 固收 / 闲钱类策略

推荐：

- 远山
- 湖面
- 低饱和蓝
- 柔和晨光
- 平缓路径

关键词：

- 安稳
- 打底
- 低波
- 慢慢变好

##### 进取 / 信号 / 权益类策略

推荐：

- 山路
- 光线
- 信号面板
- 节奏点
- 数据卡片

关键词：

- 节奏
- 机会
- 纪律
- 信号
- 全球配置

##### 长期配置 / 多资产策略

推荐：

- 山海
- 道路
- 罗盘
- 资产配置环
- 地平线

关键词：

- 长期
- 分散
- 配置
- 穿越周期

##### 现金流 / 定取策略

推荐：

- 水流
- 果树
- 光照
- 收获
- 柔和暖色点缀

关键词：

- 生长
- 收获
- 现金流
- 陪伴

#### 7. Hero Scene Completeness Check

生成或验收 Hero 时，必须检查：

- 是否有远景、中景、前景？
- 是否有明确光源？
- 背景元素是否有场景语义？
- 右侧 icon 是否像贴上去的？
- icon 是否足够精细？
- Hero 底部是否与首卡自然衔接？
- 整体是否像一张完整营销氛围图？
- 是否避免了单一浅色背景和抽象波浪堆叠？

如果以上检查不通过，应重新生成 Hero，不应继续沿用当前视觉。

### 6.4.10 Strategy Hero Must Not Be Empty

策略介绍页 Hero 不得为空背景。即使没有真实插画素材，也必须生成完整场景型 SVG 或精细化主视觉。

#### 强约束

策略介绍页 Hero 必须至少包含以下 4 类要素中的 3 类：

1. 远景层  
   - 天空、远山、日出、云雾、光源、地平线

2. 中景层  
   - 水面、路径、山谷、波纹、道路、层叠山形

3. 前景层  
   - 策略信号面板、资产配置卡、数据组件、精细化 icon、顾问人物

4. 过渡层  
   - 底部雾化、渐隐遮罩、首卡上浮、光影延展

如果 Hero 只有：

- 浅蓝渐变
- 抽象波浪色块
- 文字
- 孤立 icon
- 单纯色块背景

则不符合策略营销页首屏要求，必须重新生成。

#### 策略页 Hero 目标

策略介绍页 Hero 应完成三件事：

1. 建立策略情绪  
   - 稳健、长期、配置、节奏、陪伴、机会

2. 建立视觉记忆点  
   - 用户看到首屏后，能记住这个策略的主题符号

3. 承接首张内容卡  
   - Hero 与首卡应形成完整的视觉路径，而不是上下断开

#### 守正出奇类页面推荐 Hero

当页面是“守正出奇 / 进取配置 / 信号发车 / 全球权益”类策略时，推荐：

- 远景：蓝色天空、远山、柔和光源
- 中景：水面或蜿蜒路径，体现长期、节奏、配置
- 前景：右侧精细化策略信号面板 icon 或资产配置小卡
- 底部：雾化渐隐，自然过渡到首张白卡

避免：

- 只有浅蓝渐变背景
- 只有抽象波浪
- 只有大标题和数据卡
- 右侧无主视觉锚点
- 图标像临时占位
- 页面像普通详情页而不是策略营销 H5

#### Hero Empty Background Check

生成后必须检查：

- Hero 是否有明确主视觉？
- Hero 是否有远景 / 中景 / 前景中的至少 2 层？
- Hero 是否有光源或空间纵深？
- Hero 是否与首卡自然衔接？
- Hero 是否能体现策略气质？
- 页面是否避免了“信息卡片页”而不是“营销 H5”的问题？

如果检查不通过，应优先重做 Hero，而不是只调整文字或卡片。

### 6.4.11 First Screen Information Density Rules

策略介绍页的首屏必须兼顾氛围感与信息获取效率。不允许为了展示 Hero 背景而出现大面积无效留白，导致用户第一屏获取信息不足。

#### 1. 首屏信息目标

在 375px 宽移动端页面中，策略介绍页首屏应尽量让用户看到以下内容中的大部分：

1. 页面标题 / 策略名称
2. 一句话策略定位
3. 2–4 个策略标签
4. 3 个左右关键指标或核心卖点
5. 首张主内容卡的上半部分或核心结论
6. 若有 CTA 锚点，可露出其引导入口

要求：

- 首屏必须同时具备“氛围感”和“信息感”。
- 不允许只有头图和标题，没有足够正文信息。
- 用户进入页面后，必须能在第一屏快速理解“这是什么策略、核心优势是什么”。
- 氛围图服务信息表达，而不是占用信息空间。

#### 2. Hero 高度控制规则

策略介绍页 Hero 虽然可以有完整氛围图，但高度必须克制。

推荐：

- Hero 总高度建议控制在 320px–400px。
- 常规策略页推荐 340px–380px。
- 除非是重氛围专题页，否则不要超过 420px。

禁止：

- Hero 过高，导致首张内容卡整体下移。
- Hero 下半部分成为纯装饰留白。
- 为了展示背景图而牺牲首屏信息密度。
- 将策略介绍页做成只有大头图的海报页。

#### 3. 首卡上浮规则

首张核心内容卡必须尽量提前进入首屏。

要求：

- 首卡应与 Hero 底部形成上浮衔接。
- 推荐上浮高度 16px–32px。
- 首卡上半部分尽量进入第一屏。
- 若页面为策略介绍页，首卡应承担“快速讲明白策略”的作用。
- 首卡可以是“一句话理解”、核心数据卡、策略定位卡或核心卖点卡。

禁止：

- 首卡完全掉出首屏。
- 首卡距离 Hero 过远。
- 首卡进入首屏后仍只露出很少一部分。
- Hero 与首卡之间出现无意义大间距。

#### 4. 中部无效留白控制

Hero 中间或底部不允许存在大面积无效空白区域。

无效留白包括：

- 只承载装饰波浪的区域。
- 没有文字、数据、图标、卡片承接的纯背景区域。
- 只为了拉开距离而存在的空白。
- 无业务含义的空旷渐变区域。

要求：

- Hero 留白应服务标题呼吸感，而不是制造信息断层。
- 装饰区域可以存在，但不能大于 Hero 高度的 25%–30%。
- 当页面存在关键内容卡时，应优先让内容进入首屏，而不是保留装饰留白。
- 如果背景路径、水面、山脉占用了大面积空间，应让其承担视觉导向或承接首卡的作用。

#### 5. 首屏信息密度平衡原则

策略介绍页首屏推荐视觉权重：

- 氛围图 / 背景：35%–45%
- 标题 / 标签 / 核心说明：25%–35%
- 首张内容卡 / 指标卡：25%–35%

目标：

- 视觉上仍有营销氛围。
- 信息上不至于像纯海报页。
- 用户能在首屏快速建立策略认知。
- 首屏既有情绪，也有可读信息。

#### 6. 守正出奇类页面的首屏推荐结构

对于“守正出奇 / 进取配置 / 信号发车 / 全球权益”类策略介绍页，首屏推荐结构：

1. 顶部导航
2. Hero 标题区
   - 策略标签
   - 策略名称
   - 一句话简介
   - 3 个标签
3. 小型主视觉 / 信号 icon
4. 3 个关键指标或 3 个核心价值点
5. 首张核心说明卡上浮进入首屏

推荐做法：

- 缩短 Hero 下半部分的纯背景区域。
- 将关键指标提前到首屏。
- 或将“一句话理解”卡片再上提。
- 让首屏既有策略情绪，又有足够信息。

#### 7. First Screen Check List

生成或验收策略页首屏时，必须检查：

- 首屏是否出现了策略名称？
- 首屏是否能看到一句话定位？
- 首屏是否露出标签或核心卖点？
- 首屏是否能看到关键指标或首张主卡的一部分？
- Hero 是否存在过多无效留白？
- 首卡是否上浮得足够合理？
- 装饰区域是否过大？
- 用户是否能在第一屏理解页面核心价值？
- 整体是否在“营销氛围”和“信息获取”之间取得平衡？

如果首屏信息不足，应优先：

1. 缩短 Hero 高度
2. 减少装饰留白
3. 提前首卡位置
4. 提前关键指标 / 核心结论出现
5. 压缩非必要背景元素

而不是继续增加纯背景区域。

## 6.5 Icon Rules

- 营销页 icon 优先使用内联 SVG。
- 不依赖外部 icon CDN 作为唯一来源。
- 风格统一：轻量、柔和、低饱和、浅蓝底。
- 不允许空白 icon、破图、emoji、粗糙线框。
- 同一页面 icon 透视、粗细、色彩必须统一。

## 6.5.1 Remix Icon Rules

且慢 UI 设计中常用 icon 组件为 **Remix Icon**。生成 App H5、功能页、策略页、活动页时，应优先从 Remix Icon 体系中选择图标，保证与现有 UI 体系一致。

### 1. 默认 icon 来源

默认 icon library：

```txt
Remix Icon
```

推荐使用方式：

```html
<i class="ri-line-chart-line"></i>
<i class="ri-shield-check-line"></i>
<i class="ri-funds-line"></i>
<i class="ri-coupon-3-line"></i>
```

或在单文件 HTML / 离线验证版本中，使用对应 Remix Icon 的 **inline SVG**。

### 2. 使用优先级

图标使用优先级：

1. Remix Icon line 风格  
   - 用于常规 UI、列表、按钮、标签、功能入口

2. Remix Icon fill 风格  
   - 用于重要状态、选中态、徽章、核心功能强调

3. 精细化营销 icon / 轻 3D icon  
   - 用于 Hero 主视觉、权益主视觉、策略主视觉、活动主视觉

4. 自定义 SVG icon  
   - 用于 Remix Icon 无法准确表达的业务概念

### 3. Remix Icon 适用场景

推荐使用 Remix Icon 的场景：

- 普通功能入口
- 步骤说明
- 列表前置 icon
- 风险提示 icon
- 数据卡辅助 icon
- 标签 icon
- CTA 辅助 icon
- 顾问服务清单 icon
- FAQ / 规则说明 icon

常用映射：

```txt
策略 / 趋势：ri-line-chart-line / ri-funds-line
配置 / 资产：ri-pie-chart-2-line / ri-stack-line
风险 / 防护：ri-shield-check-line / ri-error-warning-line
定投 / 时间：ri-calendar-check-line / ri-time-line
优惠券 / 活动：ri-coupon-3-line / ri-gift-line
账户 / 用户：ri-user-3-line / ri-account-circle-line
顾问 / 服务：ri-customer-service-2-line / ri-service-line
AI / 智能：ri-robot-2-line / ri-sparkling-line
搜索 / 诊断：ri-search-eye-line / ri-microscope-line
转托管 / 转换：ri-exchange-funds-line / ri-arrow-left-right-line
```

### 4. 营销 H5 中的 Remix Icon 风格要求

使用 Remix Icon 时，需要做轻营销化处理，而不是直接裸用黑色线框。

推荐样式：

```css
.icon-badge {
  width: 32px;
  height: 32px;
  border-radius: 12px;
  display: inline-flex;
  align-items: center;
  justify-content: center;
  background: #F0F7FF;
  color: #1B88EE;
  font-size: 18px;
}
```

要求：

- icon 颜色优先使用品牌蓝 `#1B88EE`
- 弱提示 icon 可使用浅蓝 / 灰蓝
- 风险 icon 可使用 `#FA6A55` / `#FA440C`
- 成功 / 完成状态可使用 `#07AD8F`
- icon 背景可使用浅蓝圆角底
- 同一模块 icon 尺寸、线宽、颜色必须统一

### 5. 单文件 HTML 的稳定性要求

营销 H5 验证版经常需要在本地或 Cursor 中直接打开，因此不能完全依赖外部 CDN。

规则：

- 如果使用 Remix Icon CDN，必须确保页面有降级方案。
- 关键 icon 不应只依赖远程字体图标。
- 对于 Hero、核心卖点、关键 CTA、风险提示等重要 icon，推荐直接使用 inline SVG。
- 不允许因 CDN 加载失败导致 icon 空白。
- 不允许出现方框乱码、破图或空白占位。

推荐：

```html
<!-- 常规页面可使用 Remix Icon 类名 -->
<i class="ri-shield-check-line" aria-hidden="true"></i>

<!-- 关键图标建议内联 SVG，保证离线可展示 -->
<svg class="icon" viewBox="0 0 24 24" aria-hidden="true">
  <path d="..." fill="currentColor"></path>
</svg>
```

### 6. Remix Icon 与营销主视觉的边界

Remix Icon 适合 UI 辅助，不适合作为所有营销主视觉。

不建议仅用 Remix Icon 作为：

- Hero 主视觉
- 策略头图主视觉
- 活动头图主视觉
- 权益礼遇主视觉
- 年度账单主视觉

这些场景需要：

- 精细化 2.5D / 轻 3D 图标
- 完整 SVG 插画
- 数据面板 / 金融装置
- 风景氛围图
- 顾问人物 / IP 形象

可以用 Remix Icon 辅助主视觉，但不能只放一个普通 Remix Icon 当头图。

### 7. 禁止事项

禁止：

- 使用 emoji 替代正式 icon
- 多套 icon 风格混用
- 黑色裸线框 icon 大面积出现
- icon 尺寸忽大忽小
- 外部 Remix Icon 加载失败后页面出现空白
- 用普通 Remix Icon 充当精细化营销主视觉

## 6.6 Illustration Color Rules

营销页插画应保持低饱和、轻质感、金融科技感。

推荐：

- 蓝色透明玻璃质感
- 浅紫蓝光效
- 柔和人物肤色和服装色
- 少量橙色金币 / 券 / 收益符号
- 白色高光和轻阴影

避免：

- 高饱和卡通色
- 大红大绿
- 低幼儿童插画感
- 复杂背景抢正文
- 插画色彩与页面背景割裂

## 6.6.1 AI Visual Reference Library Rules

且慢日常设计中沉淀的 AI 3D 素材、任务类插图、icon 类素材、金融装置、氛围图和人物素材，可以作为营销 H5 主视觉与模块插图的风格参考来源。

本规则只描述使用原则和风格方向，不在 Skill 中嵌入外部图片、Figma 链接或具体素材文件。

### 1. 适用范围

内部 AI 视觉素材库可作为以下场景的生成参考：

- Hero 主视觉
- 营销氛围图
- 策略介绍页头图
- 活动页权益视觉
- 3D icon / 2.5D icon
- 任务类步骤插图
- 金融数据装置
- 功能说明配图
- 账户诊断 / 持仓分析图
- 权益礼遇类视觉
- AI / 智能投顾类视觉
- 顾问陪伴 / 服务类视觉

### 2. 素材类型认知

生成视觉时，可优先从以下内部素材类型中寻找对应方向：

#### 任务类

用于：

- 步骤说明
- 任务流程
- 操作引导
- 新手指南
- 活动参与路径
- 功能使用教程

视觉特征：

- 结构清晰
- 步骤感强
- 轻 3D / 轻插画
- 不复杂、不抢正文

#### icon 类

用于：

- 功能入口
- 模块标题
- 卖点卡片
- 权益卡片
- 诊断维度
- 策略亮点

视觉特征：

- 精细化
- 低饱和
- 蓝色系为主
- 可带轻微体积感、阴影、玻璃质感
- 不使用粗糙线框或 emoji

#### 装置类

用于：

- Hero 主视觉
- 金融数据看板
- 策略信号面板
- 资产配置模型
- 收益曲线装置
- 智能工具说明

视觉特征：

- 金融属性明确
- 数据面板、曲线、卡片、环图、仪表盘等元素组合
- 轻科技感
- 有空间层次和光影

#### 氛围类

用于：

- 策略介绍页头图
- 长期配置主题
- 稳健安心主题
- 定投生长主题
- 现金流 / 收获主题

视觉特征：

- 山、水、路径、天空、光源、远山
- 低饱和、舒适、可信
- 有远景 / 中景 / 前景层次
- 与首张卡片自然衔接

#### 人物类

用于：

- 顾问陪伴
- 新手引导
- AI 助手
- 投资者场景
- 市场解读
- 账户诊断

视觉特征：

- 亲和
- 专业
- 不低幼
- 人物比例自然
- 动作服务页面信息表达

### 3. 生成优先级

当用户没有明确指定其他风格时，营销 H5 视觉生成优先级为：

1. 遵循且慢品牌视觉与 qieman h5 design 规范
2. 参考内部 AI 3D 素材库的风格语言
3. 参考用户提供的当前设计稿或历史样例
4. 根据页面 Pattern 生成对应主视觉
5. 最后才进行自由发挥

强约束：

- 不得完全脱离且慢已有视觉体系自由生成。
- 不得把内部素材风格理解为“越 3D 越好”。
- 视觉必须服务页面目标、首屏信息密度和用户理解路径。

### 4. 风格目标

生成营销图、氛围图和 icon 时，应尽量保持以下特征：

- 轻 3D / 2.5D 质感
- 蓝色或浅蓝紫为主
- 低饱和、柔和、可信
- 金融属性明确
- 图标 / 装置精细化
- 干净、克制、有呼吸感
- 与白卡内容区协调
- 不低幼、不廉价、不电商化
- 适合且慢 App 内展示

### 5. 不同页面的素材使用方向

#### 策略介绍页

优先使用：

- 氛围类风景
- 金融装置
- 策略信号面板
- 资产配置卡
- 轻 3D icon

目标：

- 建立策略气质
- 体现长期、稳健、节奏、配置
- 不要只放一个普通 UI icon

#### 活动营销页

优先使用：

- 权益 icon
- 礼盒 / 优惠券 / 金币的小型精细化视觉
- 活动步骤插图
- 轻 3D 福利装置

目标：

- 有活动感但不电商化
- 福利点清楚
- 主视觉不压过规则和 CTA

#### 定投教育页

优先使用：

- 生长 / 播种 / 浇灌视觉
- 路径 / 时间轴
- 定投卡片
- 策略小组件

目标：

- 体现长期、规律、陪伴、节奏
- 不做儿童绘本化

#### 账户诊断页

优先使用：

- 数据面板
- 扫描框
- 风险提示卡
- 账户健康装置
- 顾问人物

目标：

- 专业、可信、分析感
- 不制造恐慌

#### 功能上线页

优先使用：

- 手机界面
- 功能组件
- 操作步骤图
- 智能装置
- 小型精细 icon

目标：

- 讲清功能价值
- 降低使用门槛
- 不做纯技术后台感

#### 权益 / 会员页

优先使用：

- 勋章
- 权益卡
- 礼遇 icon
- 轻质感会员装置

目标：

- 专属感、礼遇感
- 不过度奢华

### 6. Remix Icon 与 AI 素材的分工

Remix Icon 用于常规 UI 辅助，AI 3D / 精细化素材用于营销主视觉。

推荐分工：

- 列表 icon：Remix Icon
- 步骤 icon：Remix Icon 或轻 3D icon
- 模块标题 icon：Remix Icon / 精细化 icon
- Hero 主视觉：AI 素材风格的氛围图、装置图、人物图
- 关键权益视觉：精细化 3D icon / 装置
- 策略主视觉：氛围图 + 金融装置
- 诊断主视觉：数据装置 + 顾问人物

禁止：

- 把普通 Remix Icon 放大当 Hero 主视觉。
- 用过于复杂的 3D 素材替代信息表达。
- 一页内混用多种不一致的 icon / 插画风格。

### 7. 一致性要求

如果页面使用内部 AI 素材库风格，则整页需要保持一致：

- icon 风格一致
- 光影方式一致
- 材质语言一致
- 配色体系一致
- 线条与圆角语言一致
- 插图与白卡风格一致
- Hero 主视觉与内容模块图标一致

避免：

- Hero 是精细 3D 风格，内容模块是粗糙线框。
- 首屏是氛围风景，正文图标变成卡通风。
- 活动主视觉非常精致，步骤 icon 极其简陋。
- 头图与正文完全像两套设计。

### 8. 生成验收清单

生成营销视觉时，必须检查：

- 视觉是否符合且慢低饱和、轻金融、可信的风格？
- 主视觉是否服务页面目标？
- 是否参考了内部 AI 素材库的风格方向？
- 图标 / 装置是否足够精细？
- 是否避免了普通 UI icon 直接放大当主视觉？
- 是否避免了低幼、廉价、电商化？
- 是否和页面背景、白卡、CTA 形成统一体系？
- 是否保留足够首屏信息密度？

## 6.7 Chart Rules

推荐：

```css
--chart-primary: #1B88EE;
--chart-blue: #69B1F4;
--chart-orange: #FF9F40;
--chart-warning: #EA9500;
--chart-red: #FA6A55;
```

- 过深网格线

### 6.7.2 Visual Smoothness Rules

图表和示意图应避免明显切角、断裂拼接、粗糙分段。

强约束：

- 仪表盘、环图、进度弧线不得出现明显切角。
- 多段色彩可以表达档位，但段与段之间应柔和过渡。
- SVG 图形优先使用 `stroke-linecap: round`、渐变描边、连续路径实现。
- 不使用粗糙 CSS 拼接造成锯齿、断层或错位。

## 6.10 Chart System for Marketing H5

本章节基于且慢图表设计样式沉淀，用于约束营销 H5、策略介绍页、报告页、诊断页、投教专题页中的图表表达。图表不是装饰元素，而是帮助用户快速理解收益、回撤、配置、对比、变化和风险。

### 6.10.1 Chart Design Principles

图表必须遵循以下原则：

1. 移动端优先
   - 默认放入 343px 内容卡片内。
   - 图表宽度必须适配手机阅读。
   - 坐标、图例、标签不得过密。
   - 图表高度应根据内容控制在 180px–320px 之间。

2. 结论优先
   - 每张图只表达一个核心结论。
   - 图表上方必须有标题。
   - 图表下方建议补一句解释或数据来源。
   - 不要把研究报告式复杂图表直接搬进 H5。

3. 克制清晰
   - 图表色彩低饱和。
   - 网格线弱化。
   - 坐标轴弱化。
   - 重点数据可用橙红色强调。
   - 不使用高饱和红绿对抗。

4. 数据可信
   - 必须注明数据来源、统计区间或测算口径。
   - 历史表现必须提示不代表未来。
   - 回测 / 测算 / 实盘必须区分。

---

### 6.10.2 Chart Container Rules

图表默认放在白色或浅色卡片中。

推荐结构：

```html
<section class="chart-card">
  <h3 class="chart-title">图表标题</h3>
  <p class="chart-desc">一句话解释图表要表达的结论</p>
  <div class="chart-body"></div>
  <p class="chart-source">数据来源：xxx，统计时间：xxxx</p>
</section>
```

推荐 CSS：

```css
.chart-card {
  width: 343px;
  margin: 0 auto 12px;
  padding: 16px;
  border-radius: 16px;
  background: #FFFFFF;
  box-sizing: border-box;
}

.chart-title {
  margin: 0 0 8px;
  font-size: 16px;
  line-height: 1.45;
  font-weight: 600;
  color: #333333;
  text-align: center;
}

.chart-desc {
  margin: 0 0 12px;
  font-size: 12px;
  line-height: 1.6;
  color: #606060;
}

.chart-source {
  margin: 8px 0 0;
  font-size: 10px;
  line-height: 1.5;
  color: #999999;
}
```

要求：

- 图表卡片必须有足够内边距，默认 12px–16px。
- 图表标题可以居中，说明文字建议左对齐。
- 图表和正文之间必须有留白。
- 数据来源文字不应过大，但必须可读。

---

### 6.10.3 Chart Color Tokens

推荐图表颜色：

```css
--chart-blue: #1B88EE;
--chart-blue-soft: #69B1F4;
--chart-blue-light: #A8D4FF;
--chart-orange: #FF9F40;
--chart-orange-soft: #FFB86B;
--chart-red: #FA6A55;
--chart-red-risk: #FA440C;
--chart-green: #07AD8F;
--chart-purple: #8E8CF6;
--chart-yellow: #F6C85F;
--chart-gray: #B8C4D6;
--chart-gray-light: #E6EEF7;
--chart-grid: #EDF2F7;
--chart-text: #606060;
--chart-text-light: #999999;
```

使用规则：

- 主策略 / 主数据：蓝色。
- 对比策略 / 基准：橙色或灰色。
- 收益强调：橙红色。
- 风险 / 警示：红色。
- 正向状态：绿色，但不大面积使用。
- 多资产类别可使用蓝、橙、黄、紫、青等低饱和组合。

避免：

- 同一图表超过 5 个高辨识颜色。
- 大面积高饱和红绿。
- 图表色彩与页面主题完全割裂。
- 过深网格线抢占视觉。

---

### 6.10.4 Line Chart Rules

适用场景：

- 净值走势
- 收益走势
- 指数对比
- 年化收益变化
- 收益率趋势
- 信号定投表现
- 长期回测曲线

视觉规则：

- 主线使用品牌蓝或橙色。
- 线宽建议 2px。
- 曲线可使用 `stroke-linecap: round` 和 `stroke-linejoin: round`。
- 网格线使用浅灰蓝 `#EDF2F7`。
- 坐标轴颜色弱化。
- 时间轴标签数量控制在 4–6 个。
- 多条线最多建议 2–3 条。
- 曲线下不默认填充大面积颜色，除非表达区间或风险带。

标注规则：

- 关键高点 / 低点可用小圆点 + 数值标注。
- 标注数量不超过 3 个。
- 重点数据可用橙色。
- 不要在曲线中密集标注每个点。

禁止：

- 线条过细导致手机上不可读。
- 时间标签过密。
- 多条线颜色过近。
- 无标题、无图例、无数据来源。

---

### 6.10.5 Bar Chart Rules

适用场景：

- 年度收益对比
- 最大回撤对比
- 用户行为对比
- 活动数据对比
- 收益分布
- 类别指标对比

视觉规则：

- 柱宽适中，建议 8px–16px。
- 柱子圆角建议 2px–4px。
- 正向数据可用蓝色 / 橙色。
- 负向数据可用浅蓝绿、灰蓝或风险红，需根据语义选择。
- 同组对比最多 2–3 根柱。
- 分组柱状图必须有清晰图例。

标签规则：

- 关键柱可直接标注数值。
- 所有柱都标注时，必须避免拥挤。
- 百分比统一保留 1–2 位小数。

禁止：

- 柱子过窄。
- 颜色过多。
- 标签重叠。
- 一张图同时表达太多维度。

---

### 6.10.6 Donut / Pie Chart Rules

适用场景：

- 资产配置比例
- 持仓结构
- 权益 / 债券 / 货币 / 商品占比
- 基金品类分布
- 用户资产分布

推荐优先使用 Donut Chart，而不是普通 Pie Chart。

视觉规则：

- 环图外径建议 96px–140px。
- 内圈留白适中，可放核心标签或总称。
- 分片数量建议 3–6 个。
- 色彩低饱和，使用蓝、橙、黄、紫、青等协调组合。
- 图例建议放右侧或下方，文字保持短句。
- 最大占比可轻微突出，但不使用过强爆炸效果。

标签规则：

- 图例必须包含类别名和占比。
- 小于 1% 的类别可合并为“其他”。
- 不建议在每个扇区上直接放长文字。

禁止：

- 扇区太多导致无法识别。
- 高饱和彩虹色。
- 3D 饼图。
- 扇区切角明显或边缘粗糙。

---

### 6.10.7 Progress / Gauge / Signal Chart Rules

适用场景：

- 市场温度
- 信号档位
- 买入倍数
- 风险等级
- 账户健康状态
- 目标完成度

视觉规则：

- 弧线或进度条必须圆润。
- 使用 `stroke-linecap: round`。
- 多段色彩需有自然过渡，不出现明显切角。
- 当前状态用圆点、指针或标签标识。
- 档位不宜超过 5 个。
- 颜色从低风险到高风险可使用蓝 → 紫 → 橙 → 红的柔和过渡。

禁止：

- 明显切角的仪表盘。
- 粗糙 CSS 拼接半圆。
- 多段弧线断裂。
- 指针与刻度错位。
- 视觉像工程占位。

---

### 6.10.8 Heatmap / Correlation Matrix Rules

适用场景：

- 大类资产相关性
- 基金相关性
- 行业相关性
- 风险暴露矩阵

视觉规则：

- 单元格使用柔和蓝红或蓝橙渐变。
- 负相关可偏蓝，正相关可偏红 / 橙。
- 中性值使用浅灰或白。
- 单元格圆角建议 4px。
- 字号必须可读，建议 10px–12px。
- 矩阵不宜超过 6×6，超过时移动端可读性下降。

说明规则：

- 必须解释颜色含义。
- 必须说明相关性不代表因果。
- 建议增加一句用户能理解的解读。

禁止：

- 高饱和红蓝强刺激。
- 过多单元格导致不可读。
- 只放矩阵不解释。

---

### 6.10.9 Table Chart Rules

适用场景：

- 策略指标对比
- 优惠券档位
- 年度表现
- 收益 / 回撤 / 持有期
- 用户分组对比
- 活动规则

视觉规则：

- 表格必须适配 343px 卡片。
- 不允许横向滚动作为默认方案。
- 列数建议 2–4 列。
- 行数建议 3–6 行。
- 表头使用浅蓝 / 浅灰底。
- 重要数字可用品牌蓝或橙红色强调。
- 行间分割线使用浅灰蓝。

移动端要求：

- 若信息过多，应拆成多张小表或指标卡。
- 不要把 PC 宽表直接压缩到手机。
- 表格文字不小于 10px。
- 关键指标应优先卡片化。

---

### 6.10.10 Data Label Rules

数据标注必须简洁、克制。

推荐：

- 标注关键高点
- 标注关键低点
- 标注当前值
- 标注目标值
- 标注活动截止点
- 标注风险阈值

规则：

- 一张图内重点标注不超过 3 个。
- 数值格式统一。
- 百分比保留 1–2 位小数。
- 金额使用 `¥`，大额可用“万 / 亿”。
- 标注颜色与语义一致。

避免：

- 所有点都标注。
- 标注遮挡曲线。
- 标注颜色与数据语义冲突。
- 小屏中出现重叠。

---

### 6.10.11 Legend Rules

图例用于帮助用户理解颜色含义，不应占据过多空间。

规则：

- 图例建议放在图表下方或右侧。
- 每个图例包含色点 / 色线 + 名称。
- 名称不宜超过 8 个汉字。
- 图例数量建议 2–5 个。
- 对比图必须有图例。
- 单系列图表可省略图例，但标题和说明必须清楚。

---

### 6.10.12 Data Source & Footnote Rules

所有图表必须有数据来源或口径说明。

推荐格式：

```txt
数据来源：且慢，统计时间：2024.01.01–2025.06.30
```

或：

```txt
数据来源：Wind，测算区间：2015.01.01–2025.06.30。历史测算不代表未来表现。
```

要求：

- 数据来源文字放在图表下方。
- 字号建议 10px–11px。
- 颜色使用 `#999999`。
- 长口径说明可放在图表卡后方或风险提示前。
- 不要把重要口径隐藏到不可读位置。

---

### 6.10.13 Chart Usage by Page Type

#### 策略介绍页

推荐图表：

- 收益走势折线图
- 最大回撤对比
- 资产配置环图
- 信号档位图
- 历史验证柱状图

重点：证明策略逻辑，而不是只展示收益。

#### 活动营销页

推荐图表：

- 收益日历
- 优惠券档位表
- 支付示例表
- 步骤流程图
- 小型进度 / 档位图

重点：降低规则理解成本。

#### 定投页

推荐图表：

- 定投 vs 一次性买入折线图
- 信号档位图
- 收益分布柱状图
- 买入倍数进度图
- 定投周期示意图

重点：解释纪律、分散和平滑波动。

#### 诊断页

推荐图表：

- 资产配置环图
- 风险提示条
- 气泡图 / 压力测试图
- 持仓集中度条形图
- 账户健康进度条

重点：让用户看到问题和优化方向。

#### 报告页

推荐图表：

- 指标网格卡
- 年度收益柱状图
- 回撤对比柱状图
- 用户行为对比表
- 功能效果对比图

重点：复盘清楚，口径严谨。

#### 投教专题页

推荐图表：

- 历史走势折线图
- 对比实验表
- 相关性矩阵
- 品类表现对比图
- 估值分位图

重点：帮助用户理解方法论。

---

### 6.10.14 Chart HTML / SVG Implementation Rules

生成单文件 HTML 时：

- 优先使用内联 SVG 或纯 HTML/CSS 绘制简单图表。
- 图表必须在离线状态下可展示。
- 不依赖 ECharts / Chart.js / D3 CDN 作为唯一方案。
- 如果使用外部图表库，必须确保页面仍有降级展示。
- 简单图表优先手写 SVG。
- 复杂图表可以使用静态 SVG 截图式实现。

SVG 推荐：

```svg
<svg viewBox="0 0 311 180" width="311" height="180" role="img" aria-label="收益走势">
  <path d="..." fill="none" stroke="#1B88EE" stroke-width="2" stroke-linecap="round" stroke-linejoin="round" />
</svg>
```

要求：

- SVG viewBox 适配 343px 卡片。
- 线条圆润。
- 图表元素命名清楚。
- 不出现锯齿、断层、明显切角。

---

### 6.10.15 Chart Acceptance Checklist

生成或验收图表时，必须检查：

- 图表是否在 343px 卡片内可读？
- 标题是否说明图表结论？
- 是否只有一个核心观点？
- 坐标轴和图例是否不过密？
- 颜色是否符合且慢低饱和体系？
- 重点数据是否被正确强调？
- 是否有数据来源和统计区间？
- 是否避免了收益承诺？
- 是否存在明显切角、断裂、拼接感？
- 离线打开时图表是否正常显示？

## 6.8 CTA Rules

- 主 CTA 默认品牌蓝。
- 按钮高度不低于 48px。
- 圆角建议 999px 或 24px。
- 文案必须是明确动作。
- 不制造焦虑，不写“错过血亏”。

## 6.9 Risk Disclosure Rules

- 风险提示必须可读。
- 字号建议 10px–12px。
- 颜色可以弱化，但不能不可读。
- 不隐藏、不折叠为唯一展示。
- 投资页、策略页、活动页均需保留相应风险提示。

## 7. Copywriting Framework

### 7.1 语气原则

- 安抚而不煽动
- 专业但不生硬
- 有陪伴感
- 有解释感
- 不承诺收益
- 不制造焦虑

### 7.2 推荐表达

- 重新出发
- 稳稳陪伴
- 用规则代替情绪

## 8. Asset Handling Rules

### 8.1 首屏图

- 首屏氛围图不得使用粗糙 CSS 占位图作为最终结果。
- 有真实头图时，应优先嵌入图片或使用可访问路径。
- 无真实头图时，应使用完整 SVG 插画，而不是多层 CSS background 拼接。

### 8.2 本地资源

- 不得引用不存在的本地相对路径图片。
- 如页面依赖本地素材文件夹，必须说明依赖关系。
- 默认输出验证版时，优先生成单文件 HTML。

### 8.3 主理人 / 顾问头像

- 有真实头像时使用真实头像。
- 无真实头像时使用低饱和虚拟头像。
- 禁止留空、破图、默认浏览器占位图。

### 8.4 产品截图

- 截图必须清晰。
- 必须高亮关键操作区域。
- 一张截图只讲一个重点。
- 不直接堆原始长图。

## 11. Document-to-H5 Compression Rules

当输入是完整文档时，不得直接把原文铺满页面。

流程：

1. 提炼一句页面主张。
2. 提炼 3 个以内关键词。
3. 提炼首屏关键信息。
4. 选择最适合的 Pattern。
5. 选择 5–8 个核心模块。
6. 每个模块只保留一个核心观点。
7. 长段落改成标题 + 短说明 + 卡片 / 图示。
8. 复杂机制改成流程图、步骤卡、示意图。
9. 数据必须补充口径。
10. 风险提示完整保留。

- 转化动作

## 13. Generation Checklist

### 13.1 页面目标

- 是否明确页面类型？
- 是否只有一个主目标？
- CTA 是否匹配目标？
- 是否选择了合适 Pattern？

### 13.2 首屏

- 用户一屏内是否知道页面在讲什么？
- 主视觉是否完整？
- 首屏与内容区是否自然过渡？
- 首屏是否没有过度挤压正文？

### 13.3 内容

- 是否按“场景 → 价值 → 证明 → 行动”递进？
- 是否每屏一个核心观点？
- 是否避免长文堆叠？
- 是否进行了移动端阅读化压缩？

### 13.4 视觉

- 策略页 Hero 是否具备远景 / 中景 / 前景和明确光源？
- 策略页 Hero 是否避免了空背景、纯渐变、抽象波浪和孤立 icon？
- 策略页 Hero 是否更像完整营销氛围图，而不是普通信息详情页？
- 策略页首屏是否避免了中部无效留白过多？
- 首张核心内容卡是否上浮并进入第一屏？
- 用户是否能在第一屏看到策略名、定位、标签和至少一组核心信息？
- Hero 是否避免了浅蓝渐变 + 抽象波浪 + 孤立 icon 的拼接感？
- 是否符合浅色氛围底 + 白卡 + 蓝色 CTA？
- icon 是否优先使用 Remix Icon 体系，并在关键位置保证离线可展示？
- icon 是否精细？
- 图表是否可读？
- 卡片层级是否清楚？
- 是否避免粗糙切角和工程占位感？

### 13.5 合规

- 收益数据是否有来源？
- 是否避免收益承诺？
- 风险提示是否可读？
- 活动规则是否清楚？
- 功能说明是否没有夸大效果？

## 15. Quick Pattern Selection Guide

```txt
如果页面核心是“讲清一个策略” → Strategy Introduction Pattern
如果页面核心是“领取福利 / 活动参与” → Activity / Campaign Pattern
如果页面核心是“定投方法 / 定投活动” → DCA Pattern
如果页面核心是“市场解读 / 投教专题” → Content / Education Pattern
如果页面核心是“新功能上线 / 工具使用” → Product / Feature Pattern
如果页面核心是“权益 / 服务 / 顾问承接” → Service / Trust / Rights Pattern
如果页面核心是“年度 / 半年度复盘” → Report Pattern
```
